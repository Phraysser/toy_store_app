# Архитектурные решения (ADR)

## ADR-001: Выбор архитектурного стиля

**Статус:** Принято

**Контекст:**
Необходимо спроектировать архитектуру программной системы, обеспечивающую разделение ответственности, тестируемость и возможность независимой разработки клиентской и серверной частей для мобильного приложения интернет-магазина игрушек.

**Решение:**
Использовать адаптированный паттерн PCMEF (Presentation-Control-Mediator-Entity-Foundation). Presentation вынесен на мобильное устройство, остальные слои размещены на сервере и взаимодействуют через REST API.

**Обоснование:**
- Чёткое разделение слоёв соответствует требованиям курсового проекта.
- Направленность зависимостей сверху вниз упрощает тестирование (моки нижележащих слоёв).
- Адаптация под клиент-сервер позволяет независимо разрабатывать мобильный клиент и серверный API.

**Последствия:**
- Необходимость поддержки версионирования API для обеспечения обратной совместимости при обновлении клиента.
- Дополнительная сложность синхронизации данных между клиентом и сервером.

## ADR-002: Выбор фреймворка для серверной части

**Статус:** Принято

**Контекст:**
Требуется реализовать серверную часть на языке Java с использованием современного фреймворка, поддерживающего REST API, ORM, безопасность.

**Решение:**
Использовать Spring Boot 3.x со следующими стартерами:
- `spring-boot-starter-web` (REST контроллеры)
- `spring-boot-starter-data-jpa` (ORM)
- `spring-boot-starter-security` (аутентификация и авторизация)
- `springdoc-openapi-starter-webmvc-ui` (документация OpenAPI)

**Обоснование:**
- Spring Boot является стандартом де-факто для Java-серверов в образовательной среде и индустрии.
- Большое сообщество, обилие документации и учебных материалов.
- Встроенная поддержка всех необходимых компонентов (JPA, Security, тестирование).

**Последствия:**
- Относительно высокий порог входа для глубокого понимания внутренних механизмов Spring.
- Необходимость изучения конфигурации Security и JPA.

## ADR-003: Выбор базы данных

**Статус:** Принято

**Контекст:**
Необходимо хранить данные о пользователях, игрушках, корзинах с поддержкой транзакций и ссылочной целостности.

**Решение:**
Использовать PostgreSQL в качестве реляционной базы данных.

**Обоснование:**
- Полная поддержка ACID-транзакций.
- Богатые возможности индексирования для быстрого поиска по категориям и названиям.
- Бесплатная, с открытым исходным кодом, легко разворачивается локально.

**Последствия:**
- Требуется установка PostgreSQL на компьютере разработчика.
- Необходимо проектировать схему с учётом нормализации.

## ADR-004: Выбор механизма аутентификации

**Статус:** Принято

**Контекст:**
Необходимо обеспечить безопасный доступ к REST API, идентифицировать пользователя и разграничивать права.

**Решение:**
Использовать JSON Web Tokens (JWT) для аутентификации. Токен выдаётся после успешного входа и передаётся в заголовке `Authorization: Bearer <token>`.

**Обоснование:**
- Stateless подход не требует хранения сессий на сервере, что упрощает масштабирование.
- Токен может содержать необходимую информацию (id пользователя, роль).
- Широкая поддержка в Spring Security.

**Последствия:**
- Необходимость безопасного хранения токена на клиенте (Encrypted SharedPreferences в Android).
- Реализация механизма обновления токенов (refresh token) для удобства пользователя.

## ADR-005: Выбор подхода к документированию API

**Статус:** Принято

**Контекст:**
Требуется предоставить документацию по REST API для последующей интеграции с мобильным клиентом.

**Решение:**
Использовать OpenAPI 3 (Swagger) с помощью библиотеки `springdoc-openapi`.

**Обоснование:**
- Автоматическая генерация интерактивной документации на основе аннотаций в коде.
- Соответствие требованиям методических указаний (траектория В).
- Удобство тестирования эндпоинтов через Swagger UI во время разработки.

**Последствия:**
- Необходимость добавления аннотаций `@Operation`, `@Schema` для улучшения читаемости документации.
